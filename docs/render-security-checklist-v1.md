# Render公開前セキュリティチェックリスト（このスレッド専用）

## 目的
Render環境での公開前に、他スレッド作業と重複しにくいよう、本番運用観点だけを一元管理する。

## 1. Render基盤設定
- [ ] サービスの `Environment` に本番用設定のみ登録
  - 必須: `NODE_ENV=production`
  - 必須: `DOCTOR_COOKIE_SECURE=true`
  - 必須: `DOCTOR_DEMO_EMAIL`, `DOCTOR_DEMO_PASSWORD` は本番用値を設定
  - 必須: `OPENAI_API_KEY` 等の外部キーを環境変数管理
- [ ] `Start Command` が `npm start` かを確認
- [ ] `Auto Deploy` の対象ブランチを明示し、意図しないブランチからの自動デプロイを防止
- [ ] `Health Check Path` は `/health` を監視し、失敗時の自動ロールバックが効くことを確認
- [ ] Render側の `Build Logs` / `Deploy Logs` を監視対象に登録

## 2. HTTPS・Cookie・セッション
- [ ] HTTPS化を前提に全通信が `https://` で配信されることを確認
- [ ] 本番Cookie設定で `Secure` と `HttpOnly` を有効化
- [ ] `SameSite` 方針（最低 `Lax`）を確認
- [ ] サーバー時刻のずれを考慮し、セッション/リンク期限テストを再実施

## 3. アクセス制御（アプリ実装外しにくい観点）
- [ ] 医師ログイン失敗時のレスポンスが想定どおり `401 / 429` に収束していること
- [ ] 共有トークン不正・失効・期限切れ時に `410/404` を返していること
- [ ] cookie失効時の `Set-Cookie` でクッキー破棄ヘッダが付与されること
- [ ] 患者系API（`user_id` 指定系）に対する将来導入予定の認証/認可設計を公開前に確定（この項目は開発タスク側で対応）

## 4. ログ・監査
- [ ] Renderのアクセスログで異常値（短時間のログイン連打、共有リンク多重発行）を可視化
- [ ] ログファイルへの過度な個人情報出力がないことを確認
- [ ] パスワード・トークン類の文字列がログ本文に直接出ない運用ルールを周知

## 5. 公開後の運用手順
- [ ] 初回デプロイ後、`/health` と主要APIを短時間で再試験
- [ ] 医師ログイン〜共有リンク発行〜コメント保存〜ログアウトを1セットで確認
- [ ] 共有リンクは期限付き運用ルールに沿って定期的に失効・再発行を実施
- [ ] 不審アクセス疑い時の手順（セッション無効化、共有リンク無効化、再デプロイ）を運用で明文化

## 6. 既知の注意事項（別スレッド依存の未完タスク）
- [ ] 患者データ保護はアプリ側改善待ち（認証・認可）
- [ ] 本リストはRender運用面中心のため、アプリ内部実装の不足は別タスクで完了させる
